# HTTP 通知投递服务

## 需求理解

### 业务场景
企业内部多个业务系统在关键事件发生时，需要调用外部系统供应商提供的 HTTP(S) API 进行通知。例如：
- 用户通过第三方广告系统引流并成功注册后，通知对应的广告系统
- 用户订阅付款成功后，通知 CRM 系统更改 Contact 状态
- 用户购买商品后，通知库存系统进行库存变更
### 设计目标
实现一个内部服务，接收业务系统提交的外部 HTTP 通知请求，并尽可能可靠地投递到目标地址。

---

### 核心要点
1. **异构性**：不同供应商的 API 请求地址、Header/Body 格式各异
2. **可靠性**：业务系统只需确保通知请求能够被稳定、可靠地送达
3. **解耦性**：业务系统不应关心外部 API 的返回值，不应被投递过程阻塞
4. **可扩展性**：系统架构需要考虑支持灵活的水平扩展

---

## 整体架构与核心设计

### 架构图
```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  业务系统   │───▶│  API网关    │───▶│  受理服务   │───▶│  RocketMQ   │
│ (外部调用方) │    │ (Controller) │    │ (Dispatch)  │    │ (消息队列)   │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
                                                              │
                                                              ▼
                                                    ┌─────────────────┐
                                                    │  MySQL         │
                                                    │ (观测层/幂等)   │
                                                    └─────────────────┘
                                                              │
                                                              ▼
                                                    ┌─────────────────┐
                                                    │  消费者         │
                                                    │ (MQConsumer)    │
                                                    └─────────────────┘
                                                              │
                                                              ▼
                                                    ┌─────────────────┐
                                                    │  外部供应商API  │
                                                    │ (HTTP目标)      │
                                                    └─────────────────┘
```

### 核心设计原则

#### 1. **异步解耦设计**
- 业务系统提交后立即返回 `202 Accepted`
- 实际 HTTP 投递由后台异步完成
- 业务系统与外部 API 响应时间完全解耦

#### 2. **可靠投递保证**
- **至少一次**：保证消息至少被投递一次
- **事务消息**：保证"DB 落库"与"MQ 入队"原子性
- **自动重试**：RocketMQ 原生 16 级延迟重试（10s→30s→1m→2m...）
- **死信队列**：重试耗尽后进入 DLQ，防止消息丢失

#### 3. **状态可观测**
- MySQL 持久化所有任务状态
- 提供任务查询接口 `GET /api/notifications/{id}`
- 定时告警 EXHAUSTED 任务

#### 4. **高吞吐架构**
- RocketMQ 消费者线程池
- 消息分区消费，支持水平扩展
- 无数据库轮询瓶颈

## 关键工程决策与取舍说明

### 决策 1：选择 RocketMQ 而非其他消息中间件
| 选项 | 优点 | 缺点 | 决策理由 |
|------|------|------|----------|
| **RocketMQ** | 原生延迟消息、事务消息、自动重试 | 运维相对复杂 | ✅ **选中**：延迟重试和事务消息直接解决核心问题 |
| Kafka | 吞吐量极高、生态完善 | 延迟较高、无原生延迟消息 | ❌ 重试机制需自行实现 |
| RabbitMQ | 延迟极低、协议丰富 | 无事务消息、大规模集群复杂 | ❌ 事务性保证不足 |

**替代方案**：如果不采用RocketMQ，可以采用高性能的Kafka，不过需要自行实现重试机制

### 决策 2：保留 MySQL 作为观测层
**问题**：RocketMQ 已保证消息不丢，为何还需要 DB？

**保留理由**：
1. **业务可观测性**：业务系统需要查询任务状态（"我的通知发出去了吗？"）
2. **幂等性校验**：防止 RocketMQ 重投导致的重复 HTTP 请求
3. **故障排查**：记录失败原因、目标 URL、请求体等上下文
4. **死信处理入口**：DLQ 消息需要关联业务信息才能人工处理


### 决策 3：不处理复杂身份认证
**支持**：
- Basic Auth（通过 Headers）
- Bearer Token（通过 Headers）
- API Key（通过 Headers）

**不支持**：
- OAuth 2.0 令牌自动刷新
- 双向 TLS 客户端证书
- AWS Signature v4 等复杂签名

**理由**：保持核心职责单一，复杂认证应由业务系统预处理后通过 Headers 传递。

### 决策 4：最大重试次数 16 次后进入 DLQ
**重试策略**：RocketMQ 默认 16 级梯度延迟
```
1次: 10s → 2次: 30s → 3次: 1m → 4次: 2m → 5次: 3m
6次: 4m → 7次: 5m → 8次: 6m → 9次: 7m → 10次: 8m
... 16次后进入 DLQ
```

**取舍**：
- ✅ **覆盖常见故障**：网络闪断、目标服务重启
- ❌ **不保证最终成功**：持续故障需人工介入
- ✅ **避免无限重试**：防止资源浪费

**告警机制**：`RetryScheduler` 每 60 秒扫描 EXHAUSTED 任务输出告警日志。

## 系统边界明确

### 本系统负责
1. ✅ 接收并持久化通知请求
2. ✅ 异步尽最大努力可靠投递到目标 URL
3. ✅ 自动重试可恢复的失败
4. ✅ 提供任务状态查询
5. ✅ 死信任务告警

### 本系统不负责
1. ❌ 解析外部 API 业务响应
2. ❌ 复杂身份认证处理
3. ❌ 保证 100% 投递成功（有 DLQ）
4. ❌ 实时推送结果给业务系统

## 扩展性考虑

### 水平扩展
1. **增加消费者实例**：多个 `NotificationMQConsumer` 实例组成 Consumer Group
2. **RocketMQ 分区**：按 `messageId` 哈希分区，提高并行度
3. **MySQL 分库分表**：随着业务规模增长，MySQL 数据库压力增大，可以考虑分库分表。

### 功能扩展
1. **多租户隔离**：DB数据隔离；资源隔离，不同租户使用不同Topic/Consumer Group
2. **批量提交**：支持 JSON 数组批量提交通知
2. **优先级队列**：不同重要性的通知进入不同 Topic
3. **回调通知**：投递完成后回调业务系统 Webhook
4. **仪表盘**：可视化展示投递统计、成功率图表

---

## 总结

本系统通过 **RocketMQ 事务消息 + MySQL 观测层** 的架构，核心价值在于：
1. **业务系统解耦**：提交即返回，不阻塞业务流程
2. **可靠性保障**：尽最大努力保障至少一次通知的语义
3. **易于扩展**：基于消息队列和MsSQL的架构易于水平扩展
